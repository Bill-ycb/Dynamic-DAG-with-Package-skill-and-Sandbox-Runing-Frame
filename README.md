# Dynamic-DAG-with-Package-skill-and-Sandbox-Runing-Frame
该架构的设计初衷是为了在宏观层面的并行性与微观层面的顺序耦合之间找到平衡。作者试图解决现有架构中的痛点：如 ReAct 模式（纯顺序执行）太慢且消耗大量 Token，而 LLMCompiler 模式（纯并行 DAG）虽然并行度高，但在节点间传递大量中间“脏数据”（如原始 HTML 或大型 JSON）会极大地消耗上下文窗口，且全局的“人类在环（HITL）”审批会不必要地阻塞其他并行的分支。

该架构的三个核心支柱如下：

1. “拆分与打包”边界规则 (The "Split vs. Package" Boundary Rule)
规划器（Planner）会根据任务的性质，动态决定应该将任务拆分为独立节点，还是打包成代码块执行：

拆分（原子节点）：对于相互独立的工具或任务（例如同时搜索 Arxiv 和 GitHub），规划器将它们拆分为并行的独立节点，由图引擎并发执行。这适用于“一对多”等发散型任务。

打包（自动打包）：对于高度耦合的任务链（例如：读取文件 -> 解析 -> 总结），规划器会将它们捆绑成一个“打包节点”（Package Node）。LLM 会生成一个临时的 Python 脚本在局部节点内依次执行这些步骤。这样可以防止中间产生的大量“脏数据”污染全局的图状态（Graph State）。

该规则实现了**“在可能的地方最大化并行，在需要顺序执行的地方最大化效率”**。

2. 沙箱反思与自我修复 (Sandbox Reflexion / Self-Healing Packages)
由于“打包节点”中包含 LLM 动态生成的代码，存在运行崩溃的风险。为此，架构引入了自我修复机制：

在执行代码包之前，图引擎会先记录状态快照。

生成的代码在一个隔离的沙箱（Sandbox）中运行。

如果运行崩溃，系统会将状态回滚，捕获报错堆栈信息（Traceback），并交由后台的一个“反思大模型（Reflexion LLM）”进行调试和代码重写，直到运行成功或达到重试次数上限。

3. 节点级人类在环与异步审批 (Node-Level HITL / Asynchronous Approval)
有别于在全局设置审批阻塞点，该架构将人类的干预（HITL）细化到了节点级别，并基于工具类别进行触发：

绿灯工具（只读）：自由执行，无需干预。

红灯工具（会改变状态的操作，如发送邮件）：会被标记为需要人类审批。

当图执行触发红灯工具时，**仅有该特定节点会被挂起（暂停）**等待人类授权，而 DAG（有向无环图）中的其他并行分支仍然可以继续异步执行，互不干扰。
